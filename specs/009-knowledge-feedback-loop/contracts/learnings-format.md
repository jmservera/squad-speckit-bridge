## Format Rules

1. **YAML frontmatter** (required): Contains machine-readable metadata delimited by `---`.
2. **Title**: `# Implementation Learnings — {Feature Name}` (H1).
3. **Metadata block**: Spec ID, execution period, agent names, sync timestamp.
4. **Six category sections** (H2): Only sections with content are included. Empty categories are omitted.
   - `## Architectural Insights`
   - `## Integration Patterns`
   - `## Performance Notes`
   - `## Decisions Made During Implementation`
   - `## Reusable Techniques`
   - `## Risks Encountered`
5. **Entry format** (H3): Title line, source attribution line (italic), content body.
6. **Attribution line**: `_Source: {attribution} ({sourceType}) | {date}_`
7. **Footer**: Generation attribution line after closing `---`.

## Content Constraints

- **Maximum entries per category**: 10 (guideline from spec edge cases). If more exist, include the 10 most recent.
- **Maximum content length per entry**: 500 characters. Longer entries are truncated with `[...]`.
- **Privacy**: All content has been through `applyPrivacyFilter()` before writing. Redaction markers like `[REDACTED:API_KEY]` may appear in content.
- **No raw transcripts**: Entries are summaries, not full agent session logs.
- **Append behavior**: On subsequent runs, new entries are appended to existing sections. Existing entries are never removed or rewritten.

## Append Semantics

When `learnings.md` already exists from a prior run:

1. Parse existing frontmatter and content.
2. Read existing entries to extract their fingerprints.
3. Append only new entries (non-duplicate fingerprints) to the appropriate category sections.
4. Update frontmatter metadata (new timestamp, cumulative counts).
5. Preserve all existing content intact.

## Machine Readability

The YAML frontmatter enables tooling to:
- Track when learnings were last updated
- Count total learnings per spec
- Detect which sources contributed
- Identify specs with no learnings (feedback loop not closed)

Tooling can parse H3 entries with the attribution pattern to extract structured data from the markdown.

## Constitution Enrichment Format

Constitution-worthy entries are appended to `.specify/memory/constitution.md` using the existing format:

```markdown
## Learnings from Spec {spec-id}

_Synced: {ISO timestamp}_

- **{title}**: {content}
```

This mirrors the existing forward sync constitution append pattern (see constitution.md lines 130-160).
