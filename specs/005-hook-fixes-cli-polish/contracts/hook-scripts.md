# Hook Script Contracts: Hook Fixes and CLI Polish (v0.3.1)

**Feature**: 005-hook-fixes-cli-polish  
**Date**: 2026-03-24  
**Status**: Complete

## Overview

The bridge exposes three lifecycle hook scripts as its primary external interface with the SpecKit pipeline. These hooks are shell scripts deployed to `.specify/extensions/squad-bridge/hooks/` and invoked by SpecKit at specific lifecycle points. This document defines the behavioral contract for each hook.

---

## Common Contract (All Hooks)

### Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `SPECKIT_SPEC_DIR` | Yes | Path to the active spec directory (set by SpecKit) |
| `BRIDGE_CONFIG` | No | Override path for bridge config file (default: `bridge.config.json`) |

### Exit Code Contract

| Exit Code | Meaning |
|-----------|---------|
| `0` | Success (including graceful skip/warning) |

**Rule**: Hooks MUST always exit `0`. They MUST NOT block the SpecKit pipeline under any circumstances. All errors are warnings that are printed to stderr/stdout and then the hook exits cleanly.

### Prerequisite Check Pattern

Every hook MUST implement the following checks in order:

1. **npx availability**: `command -v npx` — warn and exit 0 if missing
2. **SPECKIT_SPEC_DIR**: Check env var is set and non-empty — warn and exit 0 if missing
3. **Config check**: Read `bridge.config.json` (or `$BRIDGE_CONFIG`) to check if this hook is enabled — silent exit 0 if disabled
4. **Hook-specific prerequisites**: Validate any required files exist

### CLI Invocation Pattern

All hooks MUST use the canonical short alias:

```bash
npx squask <command> [args] [options]
```

**NOT** `npx @jmservera/squad-speckit-bridge` (scoped name)  
**NOT** `npx squad-speckit-bridge` (unscoped full name)

### Error Handling Pattern

```bash
npx squask <command> [args] || {
  echo "[squad-bridge] WARNING: <action> failed — <fallback message>."
  exit 0
}
```

### File Permissions

All hook template files MUST be distributed with executable permissions (`0o755` / `rwxr-xr-x`). This is enforced at two levels:
1. **Source**: Git tracks the executable bit (`git update-index --chmod=+x`)
2. **Deployment**: `FileSystemDeployer.deployExecutable()` calls `chmod(0o755)` after writing

## Hook: before-specify

**Trigger**: SpecKit invokes before specification generation (`/speckit.specify`)  
**Purpose**: Inject Squad team context into the spec directory so planning benefits from accumulated knowledge  
**Config flag**: `hooks.beforeSpecify`

### Behavioral Contract

```
INPUT:
  SPECKIT_SPEC_DIR → path to spec directory

BEHAVIOR:
  1. Check prerequisites (npx, SPECKIT_SPEC_DIR, config)
  2. Call: npx squask context "$SPEC_DIR" --quiet
  3. On success: print confirmation
  4. On failure: print warning, exit 0

OUTPUT:
  ${SPECKIT_SPEC_DIR}/squad-context.md (generated by context command)

SIDE EFFECTS:
  - Reads from .squad/ directory (skills, decisions, agent histories)
  - Writes squad-context.md to the spec directory
```

## Hook: after-tasks

**Trigger**: SpecKit invokes after task generation (`/speckit.tasks`)  
**Purpose**: Automatically create GitHub issues from generated tasks (the SpecKit→Squad handoff)  
**Config flags**: `hooks.afterTasks` (deployment), `hooks.autoCreateIssues` (runtime behavior)

### Behavioral Contract

```
INPUT:
  SPECKIT_SPEC_DIR → path to spec directory
  ${SPECKIT_SPEC_DIR}/tasks.md → generated tasks file

BEHAVIOR:
  1. Check prerequisites (npx, SPECKIT_SPEC_DIR, config)
  2. Validate tasks.md exists at ${SPECKIT_SPEC_DIR}/tasks.md
  3. Generate squad-context.md if not already present (call context command)
  4. Check hooks.autoCreateIssues config flag:
     - If false: print skip message, exit 0
     - If true (default): proceed to issue creation
  5. Call: npx squask issues "$TASKS_FILE"
  6. On success: print confirmation with issue count
  7. On failure: print warning with manual command hint, exit 0

OUTPUT:
  GitHub issues created (via issues command)
  ${SPECKIT_SPEC_DIR}/squad-context.md (if not already present)

SIDE EFFECTS:
  - Creates GitHub issues via GitHub API
  - Reads bridge configuration for labels and repository
  - Writes squad-context.md if missing
```

### Configuration Interaction

```json
{
  "hooks": {
    "afterTasks": true,           // Controls whether hook is deployed
    "autoCreateIssues": true      // Controls whether hook creates issues (default: true)
  }
}
```

When `autoCreateIssues` is `false`, the hook still runs but skips the issue creation step. It may still generate squad-context.md.

### Idempotency

The `issues` command has built-in deduplication — it checks for existing GitHub issues with matching task IDs before creating new ones. Re-running the hook after a partial failure (e.g., network timeout mid-batch) will safely create only the remaining issues.

## Hook: after-implement

**Trigger**: SpecKit invokes after implementation is complete (`/speckit.implement`)  
**Purpose**: Sync execution results and learnings back to Squad's memory for future planning cycles  
**Config flag**: `hooks.afterImplement`

### Behavioral Contract

```
INPUT:
  SPECKIT_SPEC_DIR → path to spec directory

BEHAVIOR:
  1. Check prerequisites (npx, SPECKIT_SPEC_DIR, config)
  2. Call: npx squask sync "$SPEC_DIR" --quiet
  3. On success: print confirmation
  4. On failure: print warning with manual command hint, exit 0

OUTPUT:
  Updated Squad memory files (decisions, agent histories)

SIDE EFFECTS:
  - Reads implementation results from spec directory
  - Writes to .squad/ directory (decisions.md, agent histories)
```

## Verification Checklist

| Contract Property | Verification Method |
|------------------|---------------------|
| All hooks exit 0 | Integration test: run each hook, assert exit code |
| All hooks use `squask` alias | Static analysis: grep for CLI invocations in templates |
| All hooks have +x permission | `ls -la` on source templates, git status check |
| after-tasks creates issues | E2E test: trigger hook, verify issues CLI is called |
| after-tasks respects autoCreateIssues=false | Integration test: set config, verify skip |
| Graceful failure on missing prerequisites | Unit test: unset SPECKIT_SPEC_DIR, verify warn+exit 0 |
| Config missing/malformed → defaults | Integration test: delete config, verify hook runs with defaults |
