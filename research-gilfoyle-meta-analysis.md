### Deviations from Ideal Spec Kit Flow

The canonical Spec Kit pipeline is: **Constitution → Specify → Plan → Tasks → Clarify → Implement**.

What actually happened:

1. **Constitution was never customized.** Still a `[PLACEHOLDER]` template. Richard noted this explicitly in plan.md: *"The project constitution is currently an uncustomized template."* He worked around it by deriving principles from the project's decisions.md.

2. **Specify phase was implicit.** The spec wasn't generated by running `/speckit.specify` in a vacuum — it was informed by ~90 minutes of prior Squad research (4 agents, 3 decision rounds). The spec consumed research outputs as input, not a blank user description.

3. **Clean Architecture was inserted between Plan and Tasks.** This is not a Spec Kit phase. Richard and Dinesh performed an architectural design session that produced a SKILL.md and modified plan.md's project structure. This knowledge then shaped the tasks phase (task ordering follows Clean Architecture layers, not Spec Kit's default domain-concern ordering).

4. **Clarify ran after Tasks, not before.** The first time through, clarify was skipped entirely. It was only run post-tasks, which Richard noted catches *different* issues — spec/task misalignment rather than spec ambiguity. After Juanma's intervention, it became standard: "always run clarify after tasks."

5. **The pipeline was re-run incrementally.** Adding US7 required surgical updates to spec, plan, tasks, and clarify — not a fresh run. Richard discovered `setup-plan.sh` overwrites plan.md on re-run and had to use `git restore`.

6. **No implement phase.** The session was pure planning — the pipeline stopped at tasks.

## 2. Friction Points We Hit

### FP-1: `setup-plan.sh` Overwrites plan.md on Re-Run

**What happened:** When Richard re-ran the pipeline to add US7, `setup-plan.sh` blew away the existing plan.md — which contained hours of work including the full Clean Architecture section. He had to `git restore` to recover it.

**Why it hurts:** The Spec Kit pipeline assumes first-run semantics. But real-world usage is iterative. You add a user story, you re-run. The setup script doesn't distinguish "first run" from "incremental update."

**Severity:** High. Data loss without version control is permanent.

### FP-2: Clarify Phase Was Initially Skipped

**What happened:** The first pass went Specify → Plan → (Architecture) → Tasks. Clarify wasn't run until after tasks were generated. Juanma then directed: "always run clarify after tasks."

**Why it hurts:** Without clarify, terminology drift ("context summary" vs "context document" vs "context summary document") persisted across 9+ spec sections. The degraded mode UX question sat unresolved. Post-tasks clarification caught spec/task misalignment (tasks missing `--verbose`, entities missing `approvalStatus` values) but earlier clarification would have caught terminology issues sooner.

**Silver lining:** Running clarify post-tasks proved valuable precisely because it catches different types of issues than pre-plan clarification would. Both timings have value.

### FP-3: Constitution Never Customized

**What happened:** The constitution at `.specify/memory/constitution.md` is still the raw template with `[PLACEHOLDER]` fields. Richard's plan.md explicitly notes this: *"Applying reasonable defaults based on the project's established patterns."*

**Why it hurts:** The constitution is supposed to be the quality gate for every Spec Kit phase. Without it, the "Constitution Check" in plan.md was a creative exercise in reverse-engineering principles from decisions.md. It worked here because the team had strong opinions, but it's fragile — a less experienced team would have no guardrails.

**Impact:** Moderate. The workaround (derive from decisions.md) was adequate but not repeatable.

### FP-4: Squad Agents Simulated Spec Kit Agent Behavior

**What happened:** Spec Kit's phases (specify, plan, tasks, clarify) are designed to be run as VS Code chat commands (`/speckit.specify`, `/speckit.plan`, etc.). Squad agents can't invoke these commands — they're in a different execution context (Copilot CLI vs VS Code Chat). Richard drove all Spec Kit phases manually, simulating what a Spec Kit agent would do by reading the templates and producing the artifacts.

**Why it hurts:** This is the core impedance mismatch. Squad agents are CLI-invokable background workers. Spec Kit agents are interactive VS Code chat participants. The bridge we're designing needs to handle exactly this gap, and we experienced it firsthand.

**Impact:** High. This is why the bridge exists.

### FP-5: Human-in-the-Loop Changed the Flow Non-Trivially

**What happened:** Juanma joined at ~10:03 (after research but before the Spec Kit pipeline started). His intervention had two effects: (1) he established himself as the escalation point for genuine ambiguity, and (2) he mandated always running clarify after tasks.

**Why it matters:** The Spec Kit pipeline is designed for single-agent execution. Squad's model is multi-agent with a human lead. The human's arrival changed the *governance model* mid-session: before Juanma, agents auto-resolved clarifications; after, they were supposed to escalate. This governance shift wasn't recorded in the constitution (which is still a template) — it was recorded in the decisions inbox as a directive.

**Impact:** Low friction in this session (Juanma's timing was good), but exposes a design gap: who decides what gets escalated?

### FP-6: Research Phase Has No Spec Kit Equivalent

**What happened:** The first 90 minutes were pure Squad-style parallel research — four agents investigating different facets simultaneously. Spec Kit has no concept of this. Its pipeline starts at "specify" and assumes the user already knows what they want.

**Why it hurts:** The research outputs were critical to specification quality. Without them, the spec would have been a guess. But there's no Spec Kit phase for "figure out what you're building before you specify it."

**Impact:** Medium. The workaround (doing research outside the pipeline) was natural, but the research artifacts aren't tracked by Spec Kit's workflow.

## 3. What Worked Well

### WW-1: Team Memory Informing Planning

The pipeline's standout moment was the Clean Architecture decision. Richard and Dinesh designed the architecture between plan and tasks. This created a SKILL.md that Richard then referenced during the tasks phase: *"Tasks follow Clean Architecture layer ordering: Entities → Use Cases → Adapters → Frameworks/Drivers (innermost first, per `.squad/skills/clean-architecture-bridge/SKILL.md`)."*

A decision made at 10:51 was shaping task generation at 11:01 — 10 minutes later. The skill mechanism carried knowledge through the pipeline. This is the knowledge flywheel working in miniature.

### WW-2: Parallel Research Before Sequential Pipeline

Running four agents in parallel for research, then feeding synthesized results into a sequential Spec Kit pipeline, was dramatically more efficient than either approach alone. Squad's parallelism handled the exploratory phase; Spec Kit's sequential discipline handled the convergent phase.

The pattern — **diverge (Squad) → converge (Spec Kit)** — is arguably the core value proposition of the bridge.

### WW-3: Ceremony Concept (Vetting Before Commitment)

The Design Review ceremony concept emerged from Richard's follow-up analysis. The principle — "no issues until the team has reviewed tasks" — was applied to our own process: we didn't jump to implementation after tasks were generated. We ran clarify, added US7, re-ran the pipeline, and only committed to the task list after multiple review passes.

This ceremony-before-commitment pattern prevented premature lock-in on an incomplete task list (the original 46 tasks grew to 51 after the docs story was added).

### WW-4: Decision Accumulation Was Manageable

By session end, `decisions.md` contained 8 major decisions totaling ~300 lines. This is well within usable context size. The drop-box pattern (agents write to inbox, Scribe merges) prevented conflicts during parallel research. Jared's warning about state accumulation (aithena's 475KB decisions.md) was instructive but didn't manifest here — we stayed lean.

### WW-5: Spec Quality Was High

The spec produced (spec.md) is genuinely good: 7 user stories with acceptance scenarios, 24 functional requirements, 9 success criteria, 6 edge cases, 5 clarified ambiguities. It's detailed enough to implement from and structured enough to validate against. The Squad research feeding into specify produced a more informed spec than either framework could have produced alone.

### WW-6: Incremental Pipeline Re-Run Worked

Adding US7 mid-stream was surgically precise: 4 spec locations updated, 1 plan section added, 5 tasks added, 3 new clarifications resolved. The Spec Kit artifact structure supported incremental extension without rebuilding from scratch (once the `setup-plan.sh` overwrite was handled via git restore).

## 4. Learnings That Should Shape the Bridge

### L-1: The Bridge Should Auto-Run Clarify After Tasks

**Observed:** Clarify was skipped the first time. Juanma mandated it as a post-tasks step. Richard's history confirms: *"Post-tasks clarification catches spec/task misalignment."*

**Bridge influence:** The `after_tasks` hook (already planned in US5) should do more than notify about Design Review. It should also trigger a clarify pass — or at least prompt the developer to run one.

**Proposed change:** Add to the `after_tasks` hook: (1) auto-run or prompt for clarify, (2) *then* notify about Design Review. The hook becomes a two-step sequence: validate → review.

**Spec impact:** Update FR-014 to include clarify triggering alongside review notification. Add acceptance scenario to US5: "after tasks are generated, the hook prompts for clarification before review."

### L-2: The Bridge Should Detect Template-Only Constitution

**Observed:** The constitution was never customized. Richard worked around it by deriving principles from decisions.md. This is a reasonable fallback, but only if you *know* the constitution is a template.

**Bridge influence:** The bridge's `status` command (T012) should check whether the constitution contains placeholder markers (`[PLACEHOLDER]`, `[PROJECT_NAME]`, `[PRINCIPLE_1_NAME]`) and warn if so. The `context` command could also inject a note: "Warning: Spec Kit constitution is uncustomized. Planning quality may be reduced."

**Proposed change:** Add to CheckStatus use case (T012): constitution template detection. Add warning to context summary output when constitution is a template. Consider: should the bridge auto-populate the constitution from decisions.md?

**Spec impact:** Add FR-025: "The bridge SHOULD detect uncustomized Spec Kit constitutions and warn the developer." Add edge case for template constitution.

### L-3: The Bridge Must Handle Incremental Pipeline Re-Runs

**Observed:** `setup-plan.sh` overwrites plan.md. Richard lost work and had to `git restore`. Real usage is iterative — you add stories, re-run phases, evolve artifacts.

**Bridge influence:** The bridge wraps Spec Kit phases with Squad memory. If the bridge triggers or assists with Spec Kit phase execution, it must handle incremental runs safely:
- Before triggering any Spec Kit phase, check if output artifacts already exist
- If they exist, warn the developer and suggest incremental mode (or `git stash`)
- Never silently overwrite artifacts containing user work

**Proposed change:** Add to the bridge's workflow wrapper (not in current scope but worth planning): pre-phase safety check. For immediate v0.1: document the `setup-plan.sh` overwrite behavior in SKILL.md so Squad agents know to warn about it.

**Spec impact:** Add to SKILL.md content (T039-T040): "Warning: `setup-plan.sh` overwrites plan.md. Always commit or stash before re-running the pipeline."

### L-4: The Memory Bridge Should Feed Workflow History, Not Just State

**Observed:** The current design feeds Squad's memory (skills, decisions, agent histories) into Spec Kit planning. But this session showed that the *workflow history* — what phases ran, what got skipped, what friction points occurred — is equally valuable. Richard's history entry about `setup-plan.sh` overwrites is exactly the kind of knowledge that should inform the next planning cycle.

**Bridge influence:** The context summary should include a "Workflow Notes" section extracted from agent histories that specifically mention Spec Kit workflow patterns (setup scripts, phase ordering, clarify timing). This is a subset of agent learnings, filtered for pipeline-relevant content.

**Proposed change:** Enhance the SummarizeContent use case (T021) with a fourth content tier: workflow observations. These are learning entries that mention Spec Kit phases, commands, or artifacts — filtered by keyword matching against Spec Kit vocabulary.

**Spec impact:** Update FR-007 to include workflow-relevant learning extraction. Add workflow notes as a section in the context summary format (research.md RC-3).

### L-5: Research Phase Needs Explicit Bridge Support

**Observed:** The first 90 minutes of this session were Squad-style parallel research. Spec Kit has no concept of pre-pipeline research. The research outputs informed the spec but aren't tracked by Spec Kit's workflow.

**Bridge influence:** The bridge should recognize that the Squad → Spec Kit pipeline has a pre-phase: research. The `context` command should optionally include a "Recent Research" section summarizing research artifacts from the current branch or feature.

**Proposed change:** Add optional research artifact scanning to BuildSquadContext (T022). Look for `research-*.md` files at repo root or in the spec directory. Include relevant summaries in the context output under "Recent Research."

**Spec impact:** Minor — this is a P3 enhancement. Add a note to FR-005 about research artifact discovery.

### L-6: A Session Retrospective Should Auto-Capture Workflow Learnings

**Observed:** This meta-analysis is being done manually, after the fact. But the learnings — "clarify should run after tasks," "setup-plan.sh overwrites plan.md," "Clean Architecture changed task ordering" — are exactly what should compound across sessions.

**Bridge influence:** Add a `retrospective` subcommand (or auto-trigger via hook) that scans the session's git history, identifies Spec Kit phase executions, and produces a structured summary of: phases run, phases skipped, artifacts produced, re-runs needed, and manual interventions. This feeds directly into the next cycle's context summary.

**Proposed change:** This is a new user story (US8?) for a future version. For v0.1: ensure the SKILL.md teaches agents to write workflow observations in their history.md entries.

**Spec impact:** Consider adding US8: "Session Retrospective" as P3. For v0.1, no spec change — handle via SKILL.md content.

### L-7: The Specify Phase Benefits Enormously From Pre-Existing Team Knowledge

**Observed:** The spec we produced was high quality because it was written after 4 agents had deeply analyzed both frameworks, proposed integration strategies, identified risks, and debated delivery mechanisms. A cold-start spec would have been dramatically worse.

**Bridge influence:** This validates the core hypothesis of the memory bridge. The `before_specify` hook gap is a real problem (Spec Kit doesn't have it). The workaround — manual `bridge context` before `speckit specify` — must be prominently documented and as frictionless as possible.

**Proposed change:** Make the `context` command the default behavior when bridge is invoked without arguments. Or: add a `bridge plan <spec-dir>` wrapper that runs `context` → `specify` → `plan` in sequence, injecting Squad context before the pipeline starts.

**Spec impact:** Consider a convenience wrapper in the CLI that chains context + Spec Kit phases. Not for v0.1 spec but worth tracking.

### L-8: Governance Transitions Need to Be Recorded

**Observed:** Juanma's mid-session directive changed who decides what. Before: agents auto-resolved. After: agents escalate genuine doubts. This governance change was recorded in the decisions inbox — not in the constitution, not in the spec.

**Bridge influence:** The bridge should detect governance-relevant decisions in Squad's decisions.md (keywords: "escalate," "human review," "approval required") and surface them prominently in the context summary. These aren't regular technical decisions — they're meta-decisions about how the team operates.

**Proposed change:** Enhance RelevanceScorer (T007) to boost relevance scores for governance-tagged decisions. Add a "Governance" section to the context summary, above regular decisions.

**Spec impact:** Minor enhancement to FR-006 — add governance decision detection.

## Appendix: The Meta Pattern

This session was a live demonstration of the very problem the bridge solves. We used Squad's multi-agent orchestration to research, analyze, and make decisions. Then we used Spec Kit's pipeline to structure those decisions into a specification, plan, and task breakdown. The knowledge accumulated by Squad agents during research directly improved the specification quality. The Design Review concept was applied to our own tasks before they were finalized.

The bridge isn't theoretical. We just experienced why it needs to exist, and we experienced where it fails. Every friction point in Section 2 is a design requirement for the bridge. Every success in Section 3 validates the core architecture. The learnings in Section 4 are concrete proposals, not speculation — they come from doing the work.

That's the point of eating your own dogfood. You don't just test the recipe — you taste the friction.
